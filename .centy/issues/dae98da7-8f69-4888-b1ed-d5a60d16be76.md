# Lint: prevent wrapper functions around library types

## Problem

Wrapper functions that simply delegate to a single library call add unnecessary indirection and boilerplate. For example:

```rust
pub fn is_git_repository(project_path: &Path) -> bool {
    Repository::discover(project_path).is_ok()
}
```

This is a thin one-liner that only exists to rename a library function. Callers can use `Repository::discover(path).is_ok()` directly.

## Proposed Lint

Add a Dylint lint that detects functions whose entire body is a single expression delegating to a library (non-local) function/method, with no transformation logic.

### Detection criteria

- Function body is a single expression (no `let` bindings, no branching)
- The expression is a method call or free-function call on a type from an external crate
- The wrapper function signature mirrors the underlying call (same argument types, same return type)

The lint should be emitted as an **error**, not a warning.

### Examples

**Error:**
```rust
pub fn is_git_repository(project_path: &Path) -> bool {
    Repository::discover(project_path).is_ok()
}

pub fn parse_config(s: &str) -> Result<Config, Error> {
    Config::from_str(s)
}
```

**Allow:**
```rust
// Has additional logic
pub fn is_git_repository(project_path: &Path) -> bool {
    let result = Repository::discover(project_path);
    log::debug!("repo discovery: {:?}", result);
    result.is_ok()
}
```
